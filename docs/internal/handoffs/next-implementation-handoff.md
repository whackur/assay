## 1. 목표

네트워크 서비스, AI 공급자, 점수 산정, 인증 또는 웹 애플리케이션을 추가하기
전에 아키텍처와 기계 판독 계약을 입증하는 가장 작은 종단 간 Assay 경로를
구축한다.

목표 결과는 다음과 같다.

```text
로컬 공개 Git 저장소
  -> 불변 소스 스냅샷
  -> 결정론적 저장소 사실
  -> 파일 분류
  -> 증거 매니페스트
  -> Assay CLI가 출력하는 스키마 유효 JSON
```

첫 번째 슬라이스는 의도적으로 공개 프로젝트 점수를 만들지 않는다. 이후의
모든 점수가 신뢰해야 할 사실, 출처, 결정성 및 경계를 확립한다.

`rust-toolchain.toml`이 툴체인 버전을 결정해야 한다.

## 3. 확정된 제품 결정

구현 스파이크에서 실행 불가능하다는 증거가 나오지 않는 한 기반 마일스톤 동안
다음 결정을 다시 논의하지 않는다.

- 도메인, 수집기, 분류기, 메트릭, CLI, API 및 워커는 Rust가 담당한다.
- PostgreSQL은 호스팅 환경의 기준 저장소지만 첫 로컬 CLI 슬라이스에는 필요하지
  않다.
- TypeScript 기반 Next.js는 향후 웹 대시보드이며 메트릭이나 점수 로직의 기준이
  아니다.
- Contribution Intelligence는 개인 단위 성과 점수를 절대 출력하지 않는다.
- Project Intelligence는 차원, 신뢰도, 버전 및 증거를 포함한 투명한
  `Assay Score`를 출력한다. `Potential`은 현재 점수와 분리한다.
- 프로젝트 평가는 작성 주체와 무관하다. AI 사용 여부는 품질 신호가 아니다.
- Agent Skill은 안정적인 CLI 또는 API 계약을 호출하며 분석을 재구현하지 않는다.
- 사람 인증에는 일반 OIDC를 사용한다. 코어 코드는 배포 환경별 ID 공급자나 외부
  사용자 데이터베이스를 알지 못한다.
- 초기 배포에는 Assay 자체 회원가입이나 결제가 없다. hakhub.net의 인증을 단일
  upstream issuer로 연결하고, 외부 `admin` 역할은 명시적 배포 정책으로만 Assay
  관리자에 매핑한다.
- Anonymous Engine과 Authenticated Engine은 같은 점수 의미를 사용하되 별도
  평가 프로필과 스냅샷으로 제공한다. 구체적인 모델 문자열은 배포 설정이다.
- 비인증 결과는 자동 공개하며 저장소별 재분석 주기는 14일이다. 인증 결과는
  비공개 미리보기로 시작하며 계정·저장소별 재분석 주기는 7일이다.
- 프로젝트 유형, 보조 유형, 태그 및 성숙도를 자동 판정하고, 목적과 기능이
  유사한 경쟁 프로젝트를 1-depth로 자동 탐색한다. 상세 후보는 5개다.
- stars, forks, 다운로드 수 및 유명세는 점수를 실질적으로 올리지 않는다.
- 저장소 코드를 설치, import, 빌드, 테스트 또는 실행하지 않는다. 공개 CI와 정적
  구성은 보고된 증거로만 사용한다.
- 첫 공개 MVP에는 GitHub 공개 저장소 웹 분석과 함께 로컬 비공개 GitHub 분석,
  loopback 브라우저 대시보드, 기록 보존 및 외부 AI 명시적 동의를 포함한다.
- 초기 공개 카탈로그는 Hermes Agent와 OpenClaw 소개 카드,
  `Recently Assayed`, `Top Assays`, 상세 비교 및 README 배지를 제공한다.
- 좋아요, 댓글, 북마크, 프로젝트 claim 및 정식 이의 제기는 후속 기능이다. 초기
  피드백은 Contact 링크로 받는다.
- OpenAI API, 로컬 Codex CLI 및 실험적 공개 Codex OAuth는 공통 평가 계약 뒤의
  공급자 어댑터다.
- AI 공급자는 증거를 인용한 루브릭 판정을 만들고, 결정론적 컴파일러가 이를
  검증하고 결합한다.
- unavailable, unsupported, partial, pending 및 insufficient 데이터는 명시적인
  상태이며 오해를 낳는 0 값으로 바뀌지 않는다.
- `.orca/`는 로컬 연구 입력이며 Git에서 무시한다.

## 4. 아키텍처 방향

첫 구현의 의존성 방향은 다음과 같다.

```text
assay-cli
  -> assay-project-intelligence
      -> assay-classifier
      -> assay-git
      -> assay-domain

assay-git, assay-classifier
  -> assay-domain

assay-domain
  -> Assay 크레이트에 의존하지 않음
```

이후의 어댑터는 안쪽을 향해 의존한다.

```text
assay-github, assay-storage, assay-ai-evaluator
  -> 도메인 포트와 project-intelligence 계약

assay-api, assay-worker, assay-codex-broker
  -> 재사용 가능 크레이트

web 및 skills/assay
  -> 공개 API 또는 CLI 계약
```

규칙:

- 도메인 크레이트에는 HTTP, 데이터베이스, CLI, 공급자 또는 UI 관심사를 두지
  않는다.
- 엔트리포인트는 구성과 전송만 담당한다.
- 시간, 파일시스템 접근, Git 실행, ID 및 네트워크 접근이 결정성이나 테스트에
  영향을 주면 좁고 명확한 포트 뒤로 주입한다.
- 원시 사실을 분류, 판정 및 점수와 분리한다.
- 공개 스키마의 버전은 내부 Rust 타입과 독립적으로 관리한다.

## 5. 첫 번째 수직 슬라이스

### 5.1 명령어

다음 초기 명령어를 구현한다.

```text
assay project analyze <local-repository>
  --revision <revision>
  --evaluator deterministic
  --format json
  --output <path|->
  --no-color
```

기본값:

- `<local-repository>`를 생략했을 때만 현재 디렉터리를 기본값으로 사용한다.
- `--revision`의 기본값은 `HEAD`이며 전체 커밋 ID로 해석한다.
- 기계 판독 출력은 stdout 또는 지정한 출력 파일로 보낸다.
- 진단과 진행 상황은 stderr로 보낸다.
- 명령어는 네트워크를 호출하지 않으며 데이터베이스가 필요하지 않다.

### 5.2 초기 사실

첫 증거 매니페스트는 다음을 포함한다.

- 이식 가능한 출력에 관련 없는 호스트 경로가 노출되지 않도록 표현한 정규화된
  로컬 저장소 경로
- 분석한 전체 커밋 ID와 커밋 시각
- 사용 가능한 경우 저장소 루트 트리 ID
- 설정된 분석 버전과 스키마 버전
- 도구 버전과 규칙 세트 해시
- 감지한 언어와 지원 언어 상태
- 경로, 언어, 크기, 콘텐츠 해시 및 카테고리를 포함한 추적 파일 목록
- README, 라이선스, 패키지 매니페스트, CI, 테스트, 문서, 마이그레이션, 의존성,
  보안, 생성 파일 및 벤더 파일 존재 여부
- Git 이력 가용성과 분석 범위
- 수집기 및 분류기 경고
- 증거 ID와 출처
- 숫자형 프로젝트 점수를 포함하지 않는 전체 실행 상태

매니페스트에 전체 소스 내용을 보관하거나 출력하지 않는다.

### 5.3 초기 지원 콘텐츠

JavaScript, TypeScript, TSX 및 Python을 우선 지원한다. 다른 언어도 파일 목록에는
남기되 사라지게 하지 않고 의미 분석 상태를 `unsupported`로 표시한다.

### 5.4 인수 예시

결정론적인 합성 저장소를 사용한다.

```sh
cargo run -p assay-cli -- \
  project analyze <fixture-repository> \
  --revision HEAD \
  --evaluator deterministic \
  --format json \
  --output -
```

다음을 만족하면 명령어 구현에 성공한 것이다.

- stdout에는 스키마에 유효한 JSON 결과 하나만 있다.
- stderr에는 진단 정보만 있다.
- 결과가 정확한 불변 리비전을 식별한다.
- 모든 사실에 안정적인 증거 ID와 출처가 있다.
- 생성, 벤더, 테스트, 문서, CI, 의존성 및 프로덕션 파일을 구분할 수 있다.
- 지원하지 않거나 사용할 수 없는 분석 상태가 명시적이다.
- 소스 텍스트, 비밀, 토큰 또는 머신별 절대 경로가 노출되지 않는다.
- 고정 시계와 동일한 픽스처로 바이트 단위까지 동일한 골든 출력을 만든다.
- 명령어가 명시된 CLI 종료 코드 동작을 따른다.

## 6. 구현 작업 순서

다음 항목을 순서대로 완료한다. 각 항목은 독립적으로 검토할 수 있을 만큼 작게
유지한다.

### FND-001 — Rust 워크스페이스 초기화

다음을 생성한다.

```text
Cargo.toml
rust-toolchain.toml
crates/assay-domain/Cargo.toml
crates/assay-git/Cargo.toml
crates/assay-classifier/Cargo.toml
crates/assay-project-intelligence/Cargo.toml
crates/assay-cli/Cargo.toml
```

초기 의존성은 최소화한다. 후보로 `serde`, `serde_json`, `thiserror`, `clap`,
`tracing`, 시간 라이브러리, URL/경로 안전 식별자 라이브러리 및 암호학적 해시
라이브러리가 있다. 이 항목에서는 Axum, SQLx, Next.js, AI SDK 또는 큐를 추가하지
않는다.

인수 기준:

- `cargo fmt --check`가 통과한다.
- `cargo clippy --workspace --all-targets --all-features -- -D warnings`가
  통과한다.
- `cargo test --workspace`가 통과한다.
- 각 크레이트가 의존성 방향을 지키는 비어 있거나 최소한의 공개 표면으로
  컴파일된다.

### FND-002 — 핵심 도메인 값 정의

다음과 같이 범위가 명확한 값부터 테스트 우선으로 시작한다.

```text
RepositorySource
RevisionId
ContentHash
EvidenceId
EvidenceSource
EvidenceStatus
AnalysisStatus
AnalysisVersion
RuleSetHash
SourceSnapshot
AnalysisManifest
Warning
Limitation
```

인수 기준:

- 실용적인 범위에서 유효하지 않거나 모호한 상태를 표현할 수 없게 한다.
- enum은 명세에 필요한 complete, partial, unavailable, unsupported,
  insufficient 및 pending 상태를 포괄한다.
- 직렬화 이름은 안정적인 영어 snake_case를 사용한다.
- 도메인 테스트에는 파일시스템, Git 프로세스, 네트워크 또는 시계가 필요하지
  않다.

### SCH-001 — 스키마 우선 계약 수립

`schemas/` 아래에 다음부터 시작하는 버전 스키마를 만든다.

```text
schemas/analysis-manifest/v1.json
schemas/project-evidence/v1.json
schemas/ai-judgment/v1.json
schemas/project-evaluation/v1.json
```

초기 슬라이스에서는 처음 두 개만 완성하면 된다. AI 판정 및 평가 스키마는
명세에서 검토한 계약 골격으로 시작해도 된다.

JSON Schema를 Rust 타입에서 생성할지, Rust 직렬화를 테스트하는 주 계약으로
삼을지 결정하고 문서화한다. 검증되지 않은 두 개의 기준을 만들지 않는다.

인수 기준:

- 골든 예시는 선언한 스키마 버전에 대해 유효성 검사를 통과한다.
- 필수 필드, enum, 알 수 없는 필드 정책 및 호환성 규칙이 명시적이다.
- 스키마 메이저 버전은 의미를 조용히 변경할 수 없다.

### FIX-001 — 결정론적 저장소 시나리오 구축

테스트 중 임시 Git 저장소를 초기화하는 픽스처 빌더를 만든다. 중첩된 `.git`
디렉터리를 커밋하지 않는다.

초기 시나리오:

1. TypeScript 프로덕션, 테스트, README, 라이선스 및 CI 파일
2. Python 프로덕션, 테스트, 패키지 메타데이터 및 문서 파일
3. lockfile 및 의존성만 바뀐 변경
4. `.gitattributes` 오버라이드가 있는 생성 및 벤더 경로
5. 포맷만 바뀐 수정
6. 이름 변경 및 이동
7. 지원 언어와 비지원 언어가 섞인 경우
8. README 또는 라이선스가 없는 경우
9. 저장소 경로와 파일명에 공백 또는 Unicode가 있는 경우

Git 구현이 허용하는 범위에서 픽스처 커밋 ID를 재현할 수 있도록 작성자, 이메일,
타임스탬프, 브랜치, 커밋 메시지 및 파일 내용을 고정한다.

### ADR-001 — Git 어댑터 선택

다음을 비교하는 범위가 제한된 스파이크를 수행한다.

- 설치된 Git CLI
- `gix`
- 여전히 유력한 후보라면 `git2`를 통한 `libgit2`

측정 항목:

- 리비전 및 트리 해석의 정확성
- 이후 필요한 이름 변경 및 이력 동작
- UTF-8이 아닌 경로와 특이한 경로 처리
- 고정된 소형 및 중형 저장소에서의 성능
- 배포 및 네이티브 의존성 비용
- 공격 표면과 명령어 주입 위험
- 테스트 용이성

결정을 `docs/architecture/` 아래에 기록한다. 사용하지 않는 스파이크 코드는
삭제한다.

### GIT-001 — 소스 스냅샷 수집 구현

선택한 어댑터를 도메인 포트 뒤에 구현한다. 정확한 리비전을 해석하고, 추적
파일을 열거하고, 콘텐츠를 안전하게 해시하며, 이력 가용성을 보고한다.

인수 기준:

- 심볼릭 링크, 서브모듈, 바이너리 파일, Unicode 경로 및 읽을 수 없는 콘텐츠의
  동작이 명시적이다.
- 사용자 입력을 셸 명령어 문자열에 삽입하지 않는다.
- 저장소 콘텐츠를 절대 실행하지 않는다.
- 오류는 소스 내용을 노출하지 않으면서 어느 소스 단계인지 식별한다.
- 픽스처 테스트가 선택한 엣지 케이스를 포괄한다.

### CLS-001 — 파일 분류 구현

다음에 대해 명시적이고 버전이 지정된 규칙부터 시작한다.

- 프로덕션 코드
- 테스트
- 문서
- CI/CD
- 인프라
- 스키마 마이그레이션
- 의존성 및 lockfile
- 보안 정책
- 구성
- 생성 코드
- 벤더 코드
- 빌드 출력 및 커버리지
- unknown

`.gitattributes`의 Linguist generated/vendored 오버라이드를 존중한다. 각 결과는
카테고리, 태그, 규칙 ID, 신뢰도 및 증거를 포함한다.

프로젝트 또는 조직별 규칙을 Rust에 구현하지 않는다. 구성 오버라이드는 이후의
버전 지정 정책 어댑터에 둔다.

### PRJ-001 — 결정론적 증거 매니페스트 조립

`assay-project-intelligence`는 직접 I/O를 수행하지 않고 스냅샷 사실과 분류
사실을 결합한다.

인수 기준:

- 안정적인 증거 ID는 불변 입력과 증거 유형에서 파생한다.
- 매니페스트는 원시 사실과 분류를 분리한다.
- unsupported, missing 및 partial 상태 전파를 테스트한다.
- 차원 점수나 총점을 꾸며내지 않는다.
- 이후 AI 인용에 충분한 출처 정보를 결과에 기록한다.

### CLI-001 — 첫 명령어 제공

첫 수직 슬라이스를 `assay-cli`로 연결한다.

인수 기준:

- 옵션과 종료 코드는 기능 명세를 따른다.
- 기계 출력과 로그는 서로 다른 스트림을 사용한다.
- `--no-color`와 비대화형 동작이 CI에서 작동한다.
- 출력은 JSON Schema 유효성 검사를 통과한다.
- 골든 출력은 고정 시계와 결정론적 픽스처를 사용한다.
- `assay capabilities --format json`은 구현된 범위를 정직하게 보고한다.

## 7. 첫 슬라이스 직후 작업

다음 항목은 첫 슬라이스에 포함하지 않되, 이를 위한 명확한 경계를 준비한다.

### ADR-002 — 의미론적 diff 엔진 스파이크

동일하게 검토한 JS/TS/Python 픽스처를 사용하여 네이티브 tree-sitter 매칭,
difftastic 기반 동작 및 GumTree를 비교한다. 정확성, 이동 및 이름 변경 동작,
포맷만 바뀐 동작, 성능, 라이선스, 배포 및 Rust 통합 비용을 측정한다.

### AIE-001 — 가짜 공급자를 사용한 AI 평가기 계약

증거 ID와 스키마가 안정된 뒤에만 `assay-ai-evaluator`를 구현한다. 인용된 루브릭
판정을 반환하는 결정론적 가짜 공급자로 시작한다. 알 수 없는 기준, 범위를 벗어난
평점, 누락된 인용, 조작된 증거 ID, 프롬프트 인젝션 콘텐츠 및 스키마에 유효하지
않은 출력을 거부하는지 테스트한다.

### AIE-002 — OpenAI API 어댑터

가짜 공급자 계약 테스트가 통과한 뒤 서버 측 API 어댑터를 추가한다. 자격 증명은
비밀 저장소에서 가져온다. 모델, 루브릭, 프롬프트, 샘플링, 사용량, 지연 시간 및
검증 상태를 기록한다. API 자격 증명을 저장소가 제어하는 실행 환경에 두지 않는다.

### SCR-001 — 결정론적 점수 컴파일러

결정론적 검사와 검증된 AI 판정을 독립적인 차원으로 컴파일한다. 충분성 및 보정
요건을 충족할 때까지 Assay Score는 unavailable 상태로 둔다. 신규 프로젝트는
검증 가능한 차원이 충분하면 낮은 신뢰도의 provisional 점수를 받을 수 있다.
Potential은 별도로 유지한다.

### GH-001 — GitHub URL 및 증분 캐시

공개 GitHub URL 정규화, 불변 리비전 해석, rate limit 처리 및 평가 키 캐싱을
추가한다. blob hash 기반 증분 분석, 모노레포 경계 및 큰 저장소 스트리밍을
포함한다. 이는 공개 웹 폼과 로컬 비공개 분석의 선행 조건이다.

### CMP-001 — 유형 판정 및 자동 경쟁 프로젝트 비교

주요 유형, 보조 유형, 태그 및 성숙도를 증거와 신뢰도로 판정한다. 해결 문제,
대상 사용자, 기능 및 기술 접근을 사용해 GitHub에서 1-depth 경쟁 후보를 자동
탐색한다. 상위 5개에 후보 선정 근거, 유사도와 차별점을 제공하고 후보에서 다시
탐색하지 않는다.

awesome 목록은 링크된 프로젝트를 분석하지 않는다. 목록 자체를 평가하고 같은
주제의 다른 curated list와 구성, 중복, 고유성 및 편집 품질을 비교한다.

### WEB-001 — 최소 GitHub URL 프런트엔드

연결된 CLI 경로가 안정된 뒤 Next.js 폼, 캐시 결과 보기, 경과 시간과 현재 단계를
표시하는 비동기 진행 화면, 점수 차원, 증거 상세 탐색 및 엔진별 결과를 구현한다.
비인증 결과는 자동 공개하고 인증 결과는 비공개 미리보기로 시작한다.

### CAT-001 — 공개 카탈로그와 공유

Hermes Agent와 OpenClaw 독립 소개 카드, `Recently Assayed`, `Top Assays`, 유형과
태그 필터, 프로젝트 상세 비교 및 엔진별 README SVG 배지를 구현한다. 일반
사용자에게 점수 편집이나 재시도 버튼을 제공하지 않는다.

### IAM-001 — hakhub.net 인증 배포

Assay 코어의 일반 OIDC 경계를 유지하면서 배포 설정에서 hakhub.net을 유일한
upstream issuer로 연결한다. Assay 전용 audience를 검증하고 자체 opaque session을
발급한다. hakhub.net의 `admin` 역할은 명시적 정책과 감사 기록을 통해서만 Assay
관리자로 매핑한다.

### LOC-001 — 로컬 비공개 GitHub 분석과 대시보드

이미 clone한 비공개 저장소와 이름으로 지정한 환경 변수의 최소 권한 PAT를
지원한다. 토큰 값은 인수, 로그, 결과, 오류 또는 원격 요청에 포함하지 않는다.
`assay serve`는 `127.0.0.1`에만 바인딩하고 공개 웹과 같은 계약과 컴포넌트로
기록을 렌더링한다.

비공개 소스의 AI 평가와 공개 경쟁 프로젝트 탐색은 기본 disabled 상태다. 사용자가
공급자와 전송 증거를 확인하고 명시적으로 동의한 경우에만 활성화한다. 로컬 기록은
누적 보존하며 로컬 운영자만 삭제, 복구 또는 purge할 수 있다.

### OPS-001 — 부분 실패와 관리자 작업

완료된 단계는 보존하고 실패 단계만 partial 또는 unavailable로 표시한다. 제한된
시스템 자동 재시도가 끝나면 일반 사용자는 재시도할 수 없다. 관리자만 실패 단계
재실행, soft delete, 복구 및 purge를 수행하며 모든 작업을 감사 로그에 기록한다.

## 8. 기반 마일스톤에서 연기한 작업

기반 마일스톤 동안 다음 작업을 시작하지 않는다.

- PostgreSQL 영속성과 마이그레이션
- 장기 실행 API 및 워커 배포
- 백그라운드 작업 큐
- 공개 숫자형 Assay Score 또는 Potential
- 대규모 비교 코퍼스 또는 독창성 점수
- 공개 카탈로그 및 편집 도구
- Codex OAuth broker
- hakhub.net OIDC 멤버십 및 비공개 미리보기
- 로컬 비공개 GitHub 저장소 분석과 대시보드
- 예약 재스캔 및 알림
- CLI 호환성 보존을 넘어선 Agent Skill 패키징
- 프로덕션 배포

이는 증거와 스키마 기반을 보호하기 위해 기반 마일스톤에서만 연기한 것이다.
공개 웹 분석, 두 엔진 프로필, 카탈로그, 자동 비교, README 배지, hakhub.net 인증,
로컬 비공개 GitHub 분석 및 로컬 대시보드는 모두 첫 공개 MVP 범위다.

첫 공개 MVP 이후로 연기하는 항목은 GitLab과 다른 forge, 호스팅 비공개 저장소
연결, 조직 단위 워크스페이스, 결제, 좋아요, 댓글, 북마크, 프로젝트 claim 및 정식
이의 제기 워크플로다.

## 9. 테스트 전략

### 9.1 테스트 계층

| 계층 | 목적 |
| --- | --- |
| 도메인 단위 | 값 불변 조건, 상태, ID 및 결정론적 직렬화 |
| 분류기 단위 | 범위가 명확한 픽스처별 규칙 또는 오버라이드 하나 |
| Git 어댑터 통합 | 실제 임시 Git 이력과 특이한 경로 |
| 스키마 계약 | 모든 골든 기계 출력의 JSON Schema 유효성 검사 |
| CLI 통합 | 스트림, 종료 코드, 인수, 출력 파일 및 no-color 동작 |
| 골든 | 고정 시계와 픽스처 리비전에서 검토한 안정적 출력 |

### 9.2 필수 품질 명령어

```sh
cargo fmt --check
cargo clippy --workspace --all-targets --all-features -- -D warnings
cargo test --workspace
```

스키마 유효성 검사와 픽스처 생성 명령어가 생기면 이 목록에 추가한다. CI는
개발자가 로컬에서 실행하는 것과 같은 명령어를 실행해야 한다.

### 9.3 TDD 정책

다음에는 red-green-refactor를 사용한다.

- 도메인 불변 조건
- 분류 규칙
- `.gitattributes` 오버라이드
- 리비전 및 경로 엣지 케이스
- 상태 전파
- 증거 ID 생성
- CLI 종료 코드 및 스트림 분리
- 스키마 호환성
- 저장소 코드가 어떤 분석 경로에서도 실행되지 않는지
- 엔진별 캐시와 14일·7일 cooldown
- partial 결과와 관리자 전용 재실행
- 비공개 소스 AI 동의 및 공개 코퍼스 제외

의미론적 diff를 검토하지 않고 골든 파일을 자동 갱신하지 않는다.

## 10. 초기 CI 및 저장소 위생

첫 CI 워크플로는 다음을 수행한다.

- pull request와 기본 브랜치 push에서 실행한다.
- 고정된 Rust 툴체인과 의존성 캐시를 사용한다.
- 포맷, clippy, 테스트 및 스키마 유효성 검사를 실행한다.
- 경고가 있으면 실패한다.
- 저장소 쓰기 권한을 사용하지 않는다.
- 외부 API 키를 사용하지 않는다.
- 소스 또는 자격 증명이 포함된 아티팩트를 업로드하지 않는다.

저장소 위생:

- 모든 공개 아티팩트는 영어로 유지한다.
- `.orca/`, 소스 클론, 캐시, 빌드 출력, 자격 증명 및 비공개 평가 데이터는
  추적하지 않는다.
- 중첩 Git 저장소를 커밋하지 않는다.
- 애플리케이션 유형에 맞는 lockfile을 커밋한다.
- 주요 경계 결정은 ADR로 기록한다.

| ID 결합 | 일반 OIDC와 로컬 권한을 이후 모듈에 분리한다. |
| 엔진 점수 혼동 | 엔진 프로필, 모델, 리비전과 평가 버전을 표시하고 결과를 덮어쓰지 않는다. |
| 비공개 소스 유출 | 로컬 기본 비활성, 명시적 AI 동의, loopback 바인딩 및 공개 코퍼스 제외를 강제한다. |
| 재분석 남용 | 엔진별 cooldown, 별도 abuse limit 및 관리자 전용 실패 재실행을 적용한다. |

## 12. 다음 구현자가 결정할 미결 사항

첫 슬라이스에 필요한 결정만 해결한다.

1. 초기 Git 어댑터로 Git CLI, `gix` 또는 `git2` 중 무엇을 사용할지
2. JSON Schema를 생성 아티팩트로 둘지 주 계약으로 둘지
3. 안정적인 증거 ID 정규화 방식과 해싱 알고리즘
4. 이식 가능한 출력과 UTF-8이 아닌 경로를 위한 경로 표현
5. 규칙 세트 해싱과 구성 정규화
6. 고정 시계와 결정론적 픽스처 전략

구체적인 배포 모델 문자열, 공개 점수 가중치, OIDC bridge 세부 동작,
데이터베이스 스키마, 웹 호스팅 및 abuse limit 수치는 이후 마일스톤에서 결정한다.

## 13. 다음 마일스톤 완료 정의

기반 마일스톤은 다음을 모두 만족해야 완료된다.

- Rust 워크스페이스가 의도한 의존성 방향으로 컴파일된다.
- 결정론적 픽스처 저장소를 종단 간 분석할 수 있다.
- CLI가 스키마에 유효한 증거 매니페스트를 출력한다.
- 원시 사실, 분류, 상태, 경고 및 출처가 서로 구분된다.
- 생성, 벤더, 의존성, 테스트, 문서, CI 및 프로덕션 파일을 올바르게 표현한다.
- 출력에 소스 텍스트, 자격 증명 또는 의도하지 않은 절대 경로가 없다.
- 동일하게 고정된 입력은 동일한 골든 출력을 만든다.
- 단위, 통합, 계약 및 CLI 테스트가 통과한다.
- 포맷 및 clippy 검사가 경고 금지 조건으로 통과한다.
- 선택한 Git 어댑터 결정을 기록한다.
- 공개 문서에 의도적인 계약 결정이 반영된다.

## 14. 권장 첫 pull request

첫 pull request는 다음으로 제한한다.

- Rust 워크스페이스 초기화
- `assay-domain` 값 타입 및 테스트
- 초기 analysis-manifest JSON Schema
- 스키마에 유효한 골든 예시 하나
- 포맷, clippy, 테스트 및 스키마 유효성 검사용 CI

첫 pull request에는 Git 수집, 분류, AI, 점수, API, 데이터베이스, 웹, 인증 또는
배포를 포함하지 않는다. 이렇게 해야 이후 모든 구현 항목의 기반이 되는 작고
검토 가능한 계약을 만들 수 있다.

## 15. 인수인계 시점 상태

2026-07-16 제품 인터뷰를 반영한 이 인수인계 시점에는 다음과 같다.

- 저장소에는 영문 명세, 한국어 제품 결정 기록, handoff 및 디렉터리 경계가 있다.
- Rust 워크스페이스, 실행 파일, 테스트, CI, 스키마 구현, 데이터베이스, API,
  웹 앱 또는 Skill 패키지는 구현되지 않았다.
- `.orca/`는 무시되며 계속 추적하지 않아야 한다.
- 이 인수인계 전에 작업 브랜치를 `origin/main`과 동기화했다.
- 이 인수인계 문서 자체는 명시적으로 커밋하고 push하기 전까지 새 미커밋
  문서다.
