## 1. 제품 정의

Assay는 오픈소스 프로젝트의 실체와 품질을 검토 가능한 증거로 가시화하고
정량화하는 프로젝트 시금석이다. 사용자가 자신의 GitHub 저장소를 입력했을 때
“내 프로젝트는 몇 점인가?”라는 호기심을 충족하고, 점수의 근거와 유사 프로젝트,
차별점 및 개선 지점을 함께 보여 줘 실제로 유용한 도구라는 인식을 만들어야 한다.

공개 커뮤니티에서는 프로젝트를 발견하고 분석 결과를 공유하는 카탈로그로
사용한다. 로컬 설치에서는 비공개 GitHub 저장소를 포함한 내부 측정 도구로
사용한다.

AI 보조 개발 도구의 사용 여부는 점수 입력이 아니다. Assay는 AI가 작성했는지를
추정하지 않고 다음과 같이 결과물에서 관찰되는 신호를 평가한다.

- 구현 실체와 README 주장 사이의 일치
- 템플릿, 기존 프로젝트 및 경쟁 프로젝트와의 유사성과 차별성
- 의미 있는 기능 변화와 현재 사용 가능성
- 테스트, CI, 정적 분석, 오류 처리, 의존성 및 보안 상태
- 문서, 라이선스, 기여 안내 및 오픈소스 사용 준비도

## 2. 점수 의미와 보정 기준

공개 결과는 호기심을 유도하는 `Assay Score`를 우선 보여 주되, 단일 숫자가 차원,
증거, 신뢰도 및 평가 버전을 대체하지 않는다.

인터뷰에서 설정한 보정 기준은 다음과 같다.

| 점수대 | 기준 |
| --- | --- |
| 약 50 | 잠깐 만들어진 뒤 방치됐고, 현재 환경과 의존성 관리가 오래됐으며, 기존 공개 프로젝트가 목적을 충분히 대체하는 프로젝트 |
| 약 80 | 독창적인 기능이 한두 개 이상 명확하고, 의미 있는 기능 변화와 유지보수가 지속되며, 유사 프로젝트 사이에서 차별성이 있는 프로젝트 |
| 약 95 | 유사 프로젝트가 드물고 독창성이 매우 높으며, 구현 실체, 테스트, CI, 정적 분석, 컨벤션, 의존성 및 보안 상태가 유형에 맞게 매우 우수한 프로젝트 |

`NousResearch/hermes-agent`는 95~100점대의 보정 기준으로, `openclaw/openclaw`는
약 85점의 보정 기준으로 사용한다. 이 값은 하드코딩하지 않는다. 동일한 엔진과
루브릭으로 계산한 결과가 기대와 크게 다르면 평가 규칙과 증거 수집을 검토하는
골든 벤치마크로 사용한다.

유명 조직, stars, forks, watchers, 다운로드 수 및 의존 프로젝트 수는 점수에
사실상 반영하지 않는다. 이 정보는 참고 메타데이터나 검색 후보 정렬에는 사용할
수 있지만 품질이나 유사도 점수를 올리지 않는다.

## 3. 점수 차원

현재 구현과 검증 가능한 상태는 `Assay Score`로, 미래 가능성은 별도의
`Potential`로 표시한다.

```text
Assay Score
  Project Substance
  Originality and Differentiation
  Engineering Rigor
    tests and CI
    static analysis and conventions
    reliability and error handling
    security and dependency health
  Open Source Readiness
  Maintenance Health

Potential
  별도 지표이며 Assay Score에 포함하지 않음
```

신규 프로젝트는 장기 유지보수 이력이 없더라도 확인 가능한 차원으로 잠정 점수를
받을 수 있다. 결과에는 `provisional` 상태와 낮은 신뢰도를 명시하고, 알 수 없는
차원을 0으로 바꾸지 않는다.

오래됐거나 커밋이 적다는 이유만으로 감점하지 않는다. 현재 환경에서 사용 가능하고
테스트와 문서가 일치하며 의존성과 보안 상태가 양호한 완성형 프로젝트는 높은
점수를 받을 수 있다. 반대로 오래된 Python, JavaScript 또는 다른 런타임과
의존성이 관리되지 않아 알려진 위험이나 호환성 문제가 방치됐다면 Engineering
Rigor에서 감점한다.

## 4. 프로젝트 유형과 비교 집단

Assay는 저장소의 README, 구조, 패키지, 배포 구성 및 이력을 사용해 주요
카테고리, 보조 카테고리, 태그와 성숙도를 자동 판정한다. 한 프로젝트는 복수 유형을
가질 수 있다.

초기 카테고리 예시는 다음과 같다.

- application
- library 또는 SDK
- CLI 또는 developer tool
- framework
- service 또는 platform
- plugin
- infrastructure 또는 DevOps
- awesome list 또는 curated resource
- dataset 또는 model
- template 또는 educational project
- research artifact 또는 proof of concept

평가 기준은 프로젝트 유형과 목적에 따라 `applicable`, `partially_applicable` 또는
`not_applicable`로 판정한다. `not_applicable`은 0점이 아니다. PoC에 프로덕션
서비스와 같은 보안 배포 기준을 강제하지 않고, 라이브러리에는 API 안정성,
호환성, 패키징, 테스트 및 의존성 건강도를 더 중요하게 평가한다.

90점짜리 CLI와 90점짜리 awesome 목록은 각각의 목적에 비추어 매우 우수하다는
공통 의미를 갖는다. 다만 경쟁 비교는 단순히 같은 카테고리끼리 하지 않는다.
해결하려는 문제, 대상 사용자, 실제 기능 및 기술적 접근이 유사한 프로젝트로
동적 비교 집단을 구성한다.

## 5. 경쟁 프로젝트 탐색

Assay가 경쟁 프로젝트를 자동으로 찾는다. 사용자가 비교 대상을 입력할 필요가
없다.

```text
분석 대상 프로젝트
  -> 목적, 기능, 기술 및 구조 특징 추출
  -> GitHub 공개 저장소에서 유사 후보 검색
  -> 상위 후보 비교
  -> 종료
```

탐색은 1-depth에서 종료한다. 발견한 경쟁 프로젝트를 다시 출발점으로 삼아 추가
프로젝트를 탐색하지 않는다.

프로젝트 상세 화면에는 가장 관련성이 높은 5개를 상세히 보여 주고, 추가 후보는
간략 목록으로 펼칠 수 있게 한다. 각 결과에는 후보 선정 근거, 신뢰도, 종합
유사도, 기능·문제·접근·구조별 유사도와 대상 프로젝트의 차별점을 표시한다.

awesome 목록이 나열한 개별 프로젝트는 분석하지 않는다. awesome 목록 자체의
구성, 설명, 갱신 이력 및 편집 품질을 평가하고, 같은 주제의 다른 awesome 목록을
경쟁 프로젝트로 찾아 항목 중복과 고유성을 비교한다.

## 6. 저장소 실행 금지

Assay는 분석 대상 저장소의 코드를 설치, 빌드, 테스트 또는 실행하지 않는다.
신뢰성과 빌드 상태는 다음과 같은 정적·공개 증거로만 평가한다.

- GitHub Actions 등 공개 CI의 최근 실행 상태
- 기본 브랜치의 검사 결과
- 빌드, 테스트 및 정적 분석 워크플로의 존재와 일관성
- 패키지 스크립트와 구성의 정합성
- 반복되는 실패와 일시적 실패의 구분

결과는 `ci_reported_passing`, `ci_reported_failing`,
`build_configuration_found`, `not_independently_executed` 및
`insufficient_evidence`처럼 증거 수준을 명확히 구분한다. Assay가 직접 빌드를
검증했다고 주장하지 않는다.

## 7. 분석 엔진과 재분석

결제 티어는 없다. 접근 상태에 따라 별도의 평가 엔진 프로필을 사용하고 각 결과를
독립적으로 보존한다.

| 프로필 | 대상 | 배포 모델 프로필 | 공개 정책 | 재분석 주기 |
| --- | --- | --- | --- | ---: |
| Anonymous Engine | 비인증 사용자 | `gpt 5.6 luna max` | 즉시 공개 | 저장소별 14일 |
| Authenticated Engine | hakhub.net 인증 사용자 | `gpt 5.6 sol high` | 비공개 미리보기 후 선택적 공개 | 계정·저장소별 7일 |

위 모델 문자열은 배포 설정의 별칭이며 코드나 공개 스키마에 하드코딩하지 않는다.
공개 API는 안정적인 엔진 프로필 ID를 사용한다.

비인증 사용자가 이미 공개 결과가 있는 같은 저장소를 제출하면 새 분석을 만들지
않고 기존 공개 결과로 이동한다. 인증 사용자는 공개 결과와 별도로 Authenticated
Engine 분석을 만들 수 있다. 리비전이 변하지 않았다면 캐시를 반환한다.

분석 기록은 저장소, 불변 리비전, 증거 추출 버전, 규칙·루브릭·컴파일러 버전 및
엔진 프로필로 식별한다. 엔진별 결과를 덮어쓰거나 섞지 않는다.

## 8. 분석 작업과 실패 처리

URL 제출은 비동기 작업을 등록하고 분석 진행 화면으로 이동한다. 화면에는 가짜
퍼센트 대신 경과 시간과 현재 단계를 표시한다.

```text
저장소 확인
리비전 고정
파일과 이력 분석
프로젝트 유형 판정
CI와 의존성 증거 수집
유사 프로젝트 탐색
AI 루브릭 평가
점수 컴파일
결과 게시
```

일부 단계가 실패해도 전체 실행을 실패시키지 않는다. 완료된 영역은 표시하고
실패한 영역만 `partial` 또는 `unavailable`로 둔다. 시스템은 제한된 자동 재시도
후 종료하며 일반 사용자에게 재시도 버튼을 제공하지 않는다. 관리자는 실패 단계만
재실행할 수 있고 해당 작업을 감사 로그에 남긴다.

큰 저장소는 Git tree와 blob 스트리밍, blob hash 캐시, 생성·벤더 파일 상세 분석
제외, 모노레포 경계 감지 및 증분 재분석으로 처리한다. 하드 제한은 안전장치이며,
제한에 도달해도 저장소 크기를 감점하지 않고 분석 범위와 `partial` 상태를
명시한다.

## 9. 공개 카탈로그와 공유

비인증 분석 결과는 완료 후 공개 커뮤니티 카탈로그에 자동 게시한다. 인증 사용자의
결과는 먼저 비공개이며 사용자가 명시적으로 게시할 수 있다.

홈 화면에는 다음을 제공한다.

- Hermes Agent와 OpenClaw의 독립 소개 카드
- `Recently Assayed`
- `Top Assays`
- 카테고리, 태그, 점수 범위 및 엔진 필터

각 프로젝트 카드는 독립적으로 소개한다. 상세 화면에서만 자동 발견한 유사
프로젝트와 비교한다. 높은 점수 목록은 평가 버전, 엔진, 신뢰도와 잠정 여부를
표시한다.

공유 기능은 공개 결과 URL과 README용 SVG 배지를 포함한다. 배지는 엔진 프로필,
점수, `provisional`, `stale` 또는 `insufficient_evidence` 상태를 명시하고 해당
상세 결과로 연결한다.

좋아요, 댓글, 북마크, 팔로우, 프로젝트 claim 및 정식 이의 제기 워크플로는 첫
MVP에서 제외한다. 프로젝트 페이지에는 Contact 또는 Report an issue 링크만 둔다.
운영자는 문의를 검토해 규칙 수정, 수동 재분석 또는 결과 숨김을 수행할 수 있다.

## 10. 인증과 관리자

Assay에는 독립 회원가입이 없다. 배포 환경에서는 hakhub.net의 인증 사용자를
Assay로 연결한다. Assay 코어는 이를 특정 공급자 구현이 아닌 구성된 OIDC issuer로
취급하며 검증된 `(issuer, subject)`로 자체 계정과 opaque session을 만든다.

권장 흐름은 hakhub.net이 Assay 전용 audience의 단기 authorization code 또는
표준 OIDC 응답을 발급하고, Assay 서버가 이를 교환·검증한 뒤 자체 세션을 발급하는
방식이다. 기존 hakhub.net JWT를 URL에 넣거나 두 사이트가 browser storage 또는
공유 쿠키를 재사용하지 않는다.

hakhub.net의 `admin` 역할은 Assay 배포 정책에서 명시적으로 Assay
`Administrator`에 매핑한다. Assay가 외부 사용자 데이터베이스를 직접 조회하지
않는다.

관리자는 다음을 수행할 수 있다.

- 분석 결과 soft delete, 복구 및 별도 purge
- 실패 단계 재실행 또는 강제 재분석
- Contact 요청 처리
- 카탈로그 결과 숨김

관리자도 점수나 증거를 직접 수정할 수 없다. 모든 관리자 작업은 감사 로그에
기록한다.

## 11. 로컬 비공개 저장소 분석

첫 공개 MVP에 로컬 비공개 GitHub 저장소 분석과 로컬 브라우저 대시보드를
포함한다. GitLab 지원은 후속 작업이다.

사용자는 이미 clone한 로컬 저장소를 분석하거나 `.env`에서 최소 읽기 권한의
GitHub PAT를 참조해 비공개 저장소를 clone 또는 fetch할 수 있다. 공개 웹 폼은
PAT를 받지 않는다. 토큰을 명령행, 로그, 결과, 오류, 데이터베이스 또는 원격
서비스로 보내지 않는다.

```text
assay project analyze ./private-repository
assay project analyze https://github.com/org/private-repository \
  --github-token-env GITHUB_TOKEN
assay serve
```

로컬 대시보드는 공개 웹과 같은 컴포넌트와 버전 스키마를 사용하되 로컬 Rust API를
바라보고 `127.0.0.1`에만 바인딩한다. 로컬 단일 운영자는 로컬 관리자로 취급한다.
분석 기록과 점수 변화는 로컬에 누적 보존하며 삭제는 관리자 작업으로만 수행한다.

비공개 소스의 AI 평가와 경쟁 프로젝트 탐색은 기본적으로 비활성이다. 화면은 같은
카드를 유지하고 해당 영역을 `disabled: user_consent_required`로 표시한다. 사용자가
전송 범위와 공급자를 확인하고 명시적으로 동의한 경우에만 필요한 최소 증거를 외부
AI에 보낸다. 비공개 소스나 파생 비교 자료를 공개 카탈로그와 비교 코퍼스에 넣지
않는다.

## 12. 첫 공개 MVP

첫 공개 MVP는 다음을 모두 포함한다.

1. GitHub 공개 저장소 웹 분석
2. 경과 시간과 단계를 보여 주는 분석 진행 화면
3. Anonymous Engine 점수, 차원, 증거 및 상위 5개 유사 프로젝트
4. 자동 공개 카탈로그, 결과 URL과 README 배지
5. hakhub.net 로그인과 Authenticated Engine 비공개 미리보기
6. 엔진별 결과와 분석 이력
7. 로컬 GitHub 공개·비공개 저장소 분석
8. PAT 환경 변수와 이미 clone한 저장소 지원
9. 로컬 브라우저 대시보드와 기록 보존
10. 비공개 저장소 AI 처리의 명시적 opt-in
11. 관리자 삭제, 복구, 실패 단계 재실행 및 감사 로그

GitLab, 커뮤니티 반응, 프로젝트 claim, 정식 이의 제기, 결제, 호스팅 서비스의
비공개 저장소 연결과 조직 단위 워크스페이스는 후속 기능이다.
